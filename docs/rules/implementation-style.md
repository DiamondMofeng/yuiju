# 代码规范

## 目标

实现代码时，优先产出容易理解、容易修改、行为可预测的代码。

这里关注的不是格式层面的规范，而是日常写业务代码、设计接口、组织流程时应遵循的实现原则。

好的实现应该具备这些特点：

- 阅读者能顺着执行路径快速看懂主流程
- 关键输入、输出、状态变化和副作用清晰可见
- 代码中的概念数量尽量少，避免引入不必要的新名词
- 配置、边界条件和兜底行为是显式的，而不是靠猜
- 同一个业务事实尽量只保留一份真相源

如果一段实现虽然“更抽象”或“更统一”，却让代码更绕、更难改、更依赖隐含约定，那通常不是好的实现。

## 核心原则

### 1. 顺着执行路径写

优先让主流程按照真实执行顺序展开，让阅读者从上往下看就能理解代码在做什么。

优先选择：

- 入口清楚的主流程
- 条件判断和后续动作就近摆放
- 在调用处直接完成短小、直白的逻辑

避免：

- 为了“结构整齐”把主流程拆得支离破碎
- 需要跨多个文件来回跳转才能理解一条简单链路
- 用中间层把原本直观的流程包起来

### 2. 概念越少越好

新增类型、字段、函数、模块时，要先确认它是否真的带来了新的业务语义。

优先选择：

- 直接使用已有概念表达当前问题
- 只在确有独立语义时新增命名
- 能删掉一个概念时优先删，而不是再补一层解释

避免：

- 一个简单流程引入多个中间对象
- 同一含义在不同层里重复命名
- 明明是临时过渡数据，却包装成长期概念

### 3. 单一真相源

同一个业务事实不应在多个字段或多个结构中重复保存，尤其不要依赖它们互相回填。

优先选择：

- 一个事实只存一份
- 其他展示字段按需派生
- 必须冗余时明确谁是真相源、谁是派生值

避免：

- 两个字段描述相近含义，却没有清晰边界
- 写入时互相兜底、读取时再猜哪份更新
- 用额外字段弥补语义不清的问题

### 4. 显式优于隐式

目录、配置、阈值、副作用和边界行为要尽量写在代码里，而不是依赖隐藏约定。

优先选择：

- 显式配置项
- 明确的参数和返回结构
- 可以从代码中直接读出的行为边界

避免：

- 隐式依赖环境或目录约定
- 不看上下文就不知道会不会写文件、改状态、调外部系统
- 用默认行为掩盖关键决策

### 5. 少量重复优于错误抽象

少量可接受的重复通常比抽得不准的“通用实现”更容易维护。

优先选择：

- 保留上下文明确、语义直白的重复
- 在调用侧保留私聊/群聊、读/写等真实业务差异
- 只抽已经稳定且确实重复的部分

避免：

- 为了去掉几行重复引入一堆参数
- 调用方为了复用失去业务语义
- 为未来也许会出现的第二种情况提前搭框架

### 6. 配置与阈值要贴近语义

涉及时间、条数、目录、模型选择等配置时，数值和命名都应能解释“它控制了什么”。

优先选择：

- 阈值与实际消费方的边界一致
- 同一类配置集中在便于发现的位置
- 注释明确说明这个值影响哪条链路

避免：

- 数值之间互相打架
- 一个阈值看起来合法，但上下游边界不对齐
- 改了配置却无法判断影响范围

### 7. 注释解释约束，不重复表面代码

注释的价值在于解释目的、边界和为什么这样做，而不是把代码翻译成自然语言。

适合写注释的情况：

- 这段逻辑背后有不明显的业务约束
- 这个阈值或分支与其他链路存在耦合
- 这里的副作用或取舍不容易从代码表面看出来

不适合写注释的情况：

- 只是重复变量赋值
- 只是把函数名换一种说法再写一遍
- 代码本身已经足够直白

## 实现建议

### 1. 先写对，再写短

实现顺序应优先保证语义正确、边界清楚，再考虑是否值得压缩代码长度。

### 2. 优先在主流程里表达关键副作用

涉及写文件、发请求、更新状态、归档、触发异步任务时，尽量让调用点出现在主流程里。

### 3. 中间数据只保留到有价值为止

如果一个中间结构只是为了把数据包起来再立即拆开，通常说明它不该存在。

### 4. 先校正语义，再调整结构

当代码让人困惑时，先看字段、函数名、职责边界是不是错了，不要急着做表面重排。

### 5. 优先修正边界不清，而不是继续加兜底

如果某个字段总要靠 fallback、双写、额外校验才能工作，通常应该先回到设计边界本身看问题。

## AI Coding 常见坏代码形态

这一节用于记录 AI Coding 工具在实现代码时容易生成的低质量形态。后续遇到真实案例时，可以继续补充更具体的表现和修正方式。

### 1. 过度抽象型

表现：

- 为单一调用点新增 helper、manager、adapter、service 或 wrapper
- 把几行直白判断拆成多个函数
- 为当前只有一种场景的逻辑提前设计 strategy、provider、registry 等结构

问题：

- 主流程被拆散，阅读者需要在多个文件或函数之间跳转
- 新增概念没有带来新的业务语义，只是把复杂度搬到别处

应该优先：

- 将短小、语义直白的逻辑保留在调用处
- 只有当一段逻辑有独立业务含义、多个稳定调用点，或留在原处会明显打断主流程时，才提炼函数或模块

### 2. 兜底掩盖边界型

表现：

- 大量使用 fallback、默认值、可选字段或宽松兼容逻辑
- 上游语义不清时，在下游不断补判断、补默认行为
- 写入时同时维护多个相近字段，读取时再猜哪一个有效

问题：

- 边界问题被隐藏，真实数据契约越来越模糊
- 同一个业务事实可能出现多份真相源

应该优先：

- 先明确输入、输出和状态边界
- 在真正负责这段语义的位置完成校验和转换
- 必须兜底时，明确说明兜底服务于哪条业务链路，以及什么时候可以删除

### 3. 改动扩散型

表现：

- 为了解决一个局部问题，顺手调整无关模块、命名、目录或类型结构
- 修改范围超过需求本身，但没有说明必要性
- 借修 bug 的机会做大范围重构

问题：

- Review 难以判断哪些改动服务当前需求
- 无关变化增加回归风险，也让后续排查更困难

应该优先：

- 只修改当前问题必需触达的文件和逻辑
- 如果发现确实需要重构，应先说明原因、范围和收益，再进入实现

### 4. 语义漂移型

表现：

- 字段名、函数名或类型名看起来合理，但和真实业务含义不一致
- 为了复用旧概念，把新语义塞进并不贴切的结构里
- 同一含义在不同层使用不同名称，或不同含义使用相近名称

问题：

- 代码表面能读通，但维护者会在业务语义上误判
- 后续需求容易沿着错误概念继续扩散

应该优先：

- 先校正命名和职责边界，再调整代码结构
- 新增概念前确认它确实代表新的业务事实，而不是旧概念的别名

### 5. 流程断裂型

表现：

- 主流程只剩多个间接调用，关键判断和副作用散落在深层函数里
- 条件判断和对应动作距离很远
- 阅读一条简单链路需要不断跳转上下文

问题：

- 代码看起来分层整齐，但真实执行路径不清楚
- 状态变化、外部调用和错误处理的位置不容易发现

应该优先：

- 按真实执行顺序组织代码
- 让关键判断、状态变化和副作用在主流程中清晰可见
- 只把真正独立、稳定、会提升理解速度的逻辑拆出去

### 6. 类型表演型

表现：

- 为局部临时数据新增复杂类型、泛型或映射结构
- 类型层很完整，但业务流程没有因此更清楚
- 为了让类型复用，反而让调用处失去具体语义

问题：

- 类型复杂度超过了业务复杂度
- 维护者需要先理解额外类型体系，才能读懂简单逻辑

应该优先：

- 直接使用贴近业务语义的数据结构
- 只在类型能表达稳定边界、减少误用或提升阅读体验时新增类型

## 常见反模式

### 1. 用包装代替设计

问题没有变简单，只是多包了一层 helper、service、manager 或 adapter。

### 2. 用冗余字段弥补语义不清

当一个字段解释不清时，又补一个相近字段来“辅助理解”，通常只会制造第二份真相源。

### 3. 只看局部，不看上下游

单点实现看起来没问题，但和上下游阈值、输入输出边界拼起来以后出现断层。

### 4. 只为未来设计

当前只有一种场景，却提前抽成可扩展框架，导致代码更重、路径更长。

### 5. 靠隐藏约定工作

目录名、文件名、配置来源、默认行为没有写清楚，阅读者只能从经验猜。

## 自查问题

提交实现前，可以先问自己：

- 这段代码是不是顺着执行路径就能看懂？
- 新增的字段、类型或函数是否真的提供了新的语义？
- 是否引入了第二份真相源？
- 关键配置、阈值和副作用是否足够显式？
- 这次抽象是在减少复杂度，还是只是把复杂度移走？
- 如果删除一个中间层，代码会不会更容易理解？
- 注释有没有解释真正重要的约束，而不是重复表面代码？

## 一句话原则

实现的方向应该是：

让代码更直接、更清楚、更可预测；  
减少不必要的概念、层级和重复真相源；  
让阅读者顺着主流程就能理解系统怎么工作。
