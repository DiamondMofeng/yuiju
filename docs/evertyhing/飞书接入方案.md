# 飞书接入方案

## 目标

`@yuiju/message` 当前主要围绕 NapCat 接入 QQ 消息。接入飞书群聊后，消息服务需要同时支持不同消息源，但上层对话、记忆和回复流程不应该感知具体平台。

本次改造目标是：

- 让飞书群聊可以作为新的消息源接入悠酱对话流程。
- 保留现有 NapCat 行为，不因为接入飞书破坏 QQ 群聊链路。
- 让消息处理主流程继续围绕接收消息、记录上下文、判断是否回复、发送回复展开。
- 让人物记忆以昵称作为查询和更新主键，不写入平台用户 ID。

## Satori Runtime

消息源接入以 Satori runtime 为边界，而不是继续套用 OneBot v11，也不是项目自己实现一套 Satori-like 内部协议。

OneBot v11 更接近 QQ/OneBot 生态，字段和 ID 语义天然偏向 QQ。飞书接入后继续使用 OneBot v11 会让飞书消息被迫转换成 QQ 风格结构，容易在 at、引用回复、字符串 ID、富文本消息等地方产生语义损耗。

Satori 的消息模型更接近跨平台聊天机器人场景。它用统一的事件和会话结构表达平台、频道、用户、群成员和消息内容，消息正文使用 XHTML-like 的 Message Element 表达。这个结构比 OneBot v11 更适合作为 `@yuiju/message` 的平台接入边界。

本项目采用 Satori runtime 与 Satori adapter 作为消息源接入层，但不把 `@yuiju/message` 暴露为对外 Satori server。Satori runtime 运行在当前消息服务进程内，负责连接平台、接收事件和发送消息；`@yuiju/message` 的业务流程消费 Satori Session 和 Message Element。

- Satori runtime 负责平台连接、事件接收和发送边界。
- NapCat / OneBot 与飞书都通过 Satori runtime adapter 接入。
- 业务层消费 Satori Session 和 Message Element，不直接依赖飞书 SDK 或 NapCat SDK。
- 会话历史、LLM prompt、记忆归档可以对 Satori Session 做最小业务投影，但这个投影不承担协议实现职责。
- 面向 LLM、记忆和日志时，将 Element 渲染为 XHTML-like content 字符串。

这样可以同时保留结构化处理能力和模型阅读友好性。LLM 看到的历史内容会接近自然聊天文本，例如 `你好 <at name="悠酱"/> 今天开心吗？`，比 JSON 消息段更容易理解。

## Satori Runtime 接入边界

消息源接入统一通过进程内 Satori runtime 完成。

`@yuiju/message` 启动时初始化 Satori runtime，并注册需要启用的平台 adapter。平台 adapter 负责连接外部平台、接收平台消息、发送回复和执行平台能力；业务流程不直接依赖飞书 SDK、NapCat SDK 或具体平台事件结构。

NapCat 接入由 Satori 侧的 NapCat/OneBot 相关 adapter 承担，负责：

- 连接 NapCat WebSocket。
- 监听私聊、群聊和通知事件。
- 调用 NapCat API 发送消息、回复消息、获取群信息和被引用消息。
- 产出 Satori 风格消息事件，交给统一消息处理流程。

飞书接入由 Satori 飞书 adapter 承担，负责：

- 连接飞书开放平台事件通道。
- 接收飞书群聊消息事件。
- 调用飞书 API 发送群聊消息或回复指定消息。
- 产出 Satori 风格消息事件，交给统一消息处理流程。

统一消息处理流程只关心 Satori Session/Event 中的会话、发送者、消息正文、引用回复和原始平台数据。平台 API 调用、事件格式转换、消息发送和必要的消息补全都留在 Satori runtime 与对应 adapter 中。

这里的关键取舍是：不再为飞书单独编写项目内平台 adapter，而是把平台接入能力收口到 Satori runtime。`@yuiju/message` 只保留对 Satori 消息事件的业务编排。

## 群聊处理流程

飞书群聊接入后，群聊主流程仍沿用当前设计：

1. 接收群消息。
2. 判断是否在允许接入的群聊范围内。
3. 将消息写入当前会话上下文。
4. 根据角色状态判断是否允许发送回复。
5. 调用 LLM 判断是否回复并生成回复内容。
6. 通过 Satori bot/session 发送回复。
7. 将机器人自己的回复也写入会话上下文。

不同平台只影响消息如何接入、如何发送，以及 Satori runtime adapter 如何表达 at 和引用回复，不改变这条业务主流程。

LLM 和记忆层消费的是渲染后的 Satori content 字符串。群聊是否回复不再由业务代码提前判断是否 at 或引用悠酱，而是让 LLM 根据 `<at />`、`<quote />` 等可读消息内容和上下文自行判断。

## 人物记忆

人物记忆的查询和更新主键统一使用昵称。

也就是说：

- 不使用 QQ 号、飞书 open_id、chat_id、user_id 等平台 ID 作为人物记忆主键。
- 不按平台隔离人物记忆。
- 不建立平台账号到现实用户的绑定表。
- 不同平台里昵称相同的人会被视为同一个人。
- 用户改昵称后，旧记忆无法自动关联，这是当前方案接受的边界。

记忆内容和 LLM 可见材料中只体现昵称、场景、时间和对话内容，不写入平台用户 ID。

## 记忆文件命名

昵称是人物记忆的业务主键，文件名只是开发者可读的物理存储路径，不参与查询和更新语义。

昵称可能包含路径分隔符、特殊字符、换行或控制字符。为了让文件系统写入稳定，同时保留文件名的可读性，人物记忆文件名只转义文件系统危险字符，中文、英文、数字和普通空格等可读字符保持原样。

规则如下：

- JSON 内容里的 `personId` 保存昵称原文，是唯一可信的业务主键。
- 文件名由昵称派生，只作为物理存储路径，不作为人物身份来源。
- 文件名保留可读字符，只转义路径分隔符、系统保留字符、换行和控制字符等危险字符。
- 读取目录时打开 JSON 文件，以 JSON 内容里的 `personId` 为准，不从文件名反推人物身份。
- 热度文件中的 key 继续使用昵称原文，因为 JSON key 可以表达任意字符串。

这样可以同时满足“昵称作为主键”“文件名安全”和“开发者可读”三个目标。

## 配置方向

消息配置需要从只面向 NapCat 扩展为支持多个消息源。

配置中应明确声明：

- NapCat 的连接信息。
- 飞书应用的连接和鉴权信息。
- 各消息源启用状态。
- 允许接入的群聊范围。
- 内部 API 和主动消息目标使用哪一个消息源。

配置仍然集中维护在根目录 `yuiju.config.ts`，不新增隐藏环境变量或分散配置来源。

## 暂不处理的内容

第一阶段只关注飞书群聊文本消息接入。

暂不处理：

- 飞书私聊。
- 飞书图片、文件、表情包发送。
- 跨平台用户自动合并。
- 昵称变更后的记忆迁移。
- 历史 QQ 记忆文件的自动迁移。
- 将 `@yuiju/message` 暴露为对外 Satori 协议服务。

这些内容可以在飞书群聊基础链路稳定后再按实际需要补充。
