# 飞书接入方案

## 目标

`@yuiju/message` 当前主要围绕 NapCat 接入 QQ 消息。接入飞书群聊后，消息服务需要同时支持不同消息源，但上层对话、记忆和回复流程不应该感知具体平台。

本次改造目标是：

- 让飞书群聊可以作为新的消息源接入悠酱对话流程。
- 保留现有 NapCat 行为，不因为接入飞书破坏 QQ 群聊链路。
- 让消息处理主流程继续围绕接收消息、记录上下文、判断是否回复、发送回复展开。
- 让人物记忆以昵称作为查询和更新主键，不写入平台用户 ID。

## 消息模型

内部消息模型参考 Satori 协议，而不是继续套用 OneBot v11。

OneBot v11 更接近 QQ/OneBot 生态，字段和 ID 语义天然偏向 QQ。飞书接入后继续使用 OneBot v11 会让飞书消息被迫转换成 QQ 风格结构，容易在 at、引用回复、字符串 ID、富文本消息等地方产生语义损耗。

Satori 的消息模型更接近跨平台聊天机器人场景。它用统一的事件和会话结构表达平台、频道、用户、群成员和消息内容，消息正文使用 XHTML-like 的 Message Element 表达。这个结构比 OneBot v11 更适合作为 `@yuiju/message` 的内部统一消息模型参考。

本项目不需要完整实现 Satori HTTP/WebSocket 协议，也不要求把 `@yuiju/message` 变成 Satori server。更合适的方向是采用 Satori 风格的轻量内部模型：

- 消息信封表达平台、消息 ID、会话、发送者、群成员、时间和原始平台数据。
- 消息正文采用 Satori Message Element 风格。
- 内部处理可使用结构化 Element 数组。
- 面向 LLM、记忆和日志时，将 Element 渲染为 XHTML-like content 字符串。

这样可以同时保留结构化处理能力和模型阅读友好性。LLM 看到的历史内容会接近自然聊天文本，例如 `你好 <at name="悠酱"/> 今天开心吗？`，比 JSON 消息段更容易理解。

## Adapter 边界

消息源接入层需要把平台事件转换成统一的 Satori 风格消息，再交给业务流程处理。

NapCat adapter 负责：

- 连接 NapCat WebSocket。
- 监听私聊、群聊和通知事件。
- 将 NapCat 事件交给统一消息处理流程。
- 调用 NapCat API 发送消息、回复消息、获取群信息和被引用消息。
- 将 NapCat 消息段转换为 Satori 风格 Element。

飞书 adapter 负责：

- 连接飞书开放平台事件通道。
- 接收飞书群聊消息事件。
- 将飞书消息转换为 Satori 风格 Element。
- 调用飞书 API 发送群聊消息或回复指定消息。

统一消息处理流程不直接依赖飞书 SDK，也不直接依赖 NapCat SDK。平台 API 调用、事件格式转换和必要的消息补全都应留在 adapter 层。

飞书接入可以评估两种实现方式：

- 使用飞书官方 SDK 自写轻量 adapter。
- 在 `@yuiju/message` 进程内嵌 Satori runtime，并使用 Satori 的飞书 adapter。

内嵌 Satori runtime 不等于单独部署一个 Satori server。它可以和当前消息服务运行在同一个 Node.js 进程里。如果该方案能稳定接收飞书消息并产出 Satori Session，可以减少飞书平台适配工作。但是否采用需要先做小范围验证，重点确认依赖体积、运行边界、事件处理方式和与现有 NapCat 链路的兼容成本。

## 群聊处理流程

飞书群聊接入后，群聊主流程仍沿用当前设计：

1. 接收群消息。
2. 判断是否在允许接入的群聊范围内。
3. 将消息写入当前会话上下文。
4. 根据角色状态判断是否允许发送回复。
5. 判断消息是否直接面向悠酱，例如 at 或引用回复。
6. 调用 LLM 判断是否回复并生成回复内容。
7. 通过当前消息源 adapter 发送回复。
8. 将机器人自己的回复也写入会话上下文。

不同平台只影响消息如何接入、如何发送、如何解析 at 和引用回复，不改变这条业务主流程。

LLM 和记忆层消费的是渲染后的 Satori content 字符串。业务判断仍优先使用结构化 Element，避免为了判断 at、引用回复、图片等语义而反复解析字符串。

## 人物记忆

人物记忆的查询和更新主键统一使用昵称。

也就是说：

- 不使用 QQ 号、飞书 open_id、chat_id、user_id 等平台 ID 作为人物记忆主键。
- 不按平台隔离人物记忆。
- 不建立平台账号到现实用户的绑定表。
- 不同平台里昵称相同的人会被视为同一个人。
- 用户改昵称后，旧记忆无法自动关联，这是当前方案接受的边界。

记忆内容和 LLM 可见材料中只体现昵称、场景、时间和对话内容，不写入平台用户 ID。

## 记忆文件命名

昵称是人物记忆的业务主键，文件名只是开发者可读的物理存储路径，不参与查询和更新语义。

昵称可能包含路径分隔符、特殊字符、换行或控制字符。为了让文件系统写入稳定，同时保留文件名的可读性，人物记忆文件名只转义文件系统危险字符，中文、英文、数字和普通空格等可读字符保持原样。

规则如下：

- JSON 内容里的 `personId` 保存昵称原文，是唯一可信的业务主键。
- 文件名由昵称派生，只作为物理存储路径，不作为人物身份来源。
- 文件名保留可读字符，只转义路径分隔符、系统保留字符、换行和控制字符等危险字符。
- 读取目录时打开 JSON 文件，以 JSON 内容里的 `personId` 为准，不从文件名反推人物身份。
- 热度文件中的 key 继续使用昵称原文，因为 JSON key 可以表达任意字符串。

这样可以同时满足“昵称作为主键”“文件名安全”和“开发者可读”三个目标。

## 配置方向

消息配置需要从只面向 NapCat 扩展为支持多个消息源。

配置中应明确声明：

- NapCat 的连接信息。
- 飞书应用的连接和鉴权信息。
- 各消息源启用状态。
- 允许接入的群聊范围。
- 内部 API 和主动消息目标使用哪一个消息源。

配置仍然集中维护在根目录 `yuiju.config.ts`，不新增隐藏环境变量或分散配置来源。

## 暂不处理的内容

第一阶段只关注飞书群聊文本消息接入。

暂不处理：

- 飞书私聊。
- 飞书图片、文件、表情包发送。
- 跨平台用户自动合并。
- 昵称变更后的记忆迁移。
- 历史 QQ 记忆文件的自动迁移。
- 将 `@yuiju/message` 暴露为对外 Satori 协议服务。

这些内容可以在飞书群聊基础链路稳定后再按实际需要补充。
