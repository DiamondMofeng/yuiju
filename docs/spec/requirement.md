# 需求

## 背景

去掉 Graphiti 后，长期记忆先不做事件检索型知识库，而是先聚焦“人物记忆”。  
目标是让悠酱稳定记住每个朋友，并在互动时直接读取这个人的完整档案。

## 方案

长期记忆只保留 `people`：

- 每个朋友对应一个独立的 md 文件。
- 需要读取某个朋友时，直接加载整份人物文件，不做向量检索。
- 人物文件是当前阶段的人物长期记忆真相源。

## 目录结构

```text
memory/
  people/
    kobayashi.md
    mika.md
```

## 人物文件内容

第一版人物文件不强调复杂模块划分，而是采用固定字段的人物信息卡形式。

每个人物文件至少包含以下字段：

- 称呼
- 喜好
- 雷区
- 最近在忙什么
- 悠酱对她（他）的态度
- 最近一次值得记住的互动
- 其他补充

示例：

```md
# kobayashi

- 称呼：小林 / 小林同学
- 喜好：安静聊天，甜食，放学后慢慢散步
- 雷区：被催得很急，在很多人面前起哄她
- 最近在忙什么：最近在准备考试，晚上也会补作业，状态有点累
- 悠酱对她的态度：觉得她是个温柔但不太会主动表达的人，愿意慢慢靠近她，也会下意识照顾她的情绪
- 最近一次值得记住的互动：昨天群里聊到考试压力时，她少见地主动说自己最近有点撑不住，悠酱安慰了她几句，她后来回了一个“谢谢”，让悠酱觉得她开始愿意透露真实想法了
- 其他补充：主要在群聊里互动，偶尔也会私聊
```

更新规则：

- 除“最近一次值得记住的互动”外，其余字段都以“当前有效结论”为主，更新时优先直接覆盖旧内容。
- “最近在忙什么”只保留当前这段时间仍有效的近况，过期内容应直接替换。
- “最近一次值得记住的互动”只保留一条最新且值得记住的互动，不追加为历史列表。
- “其他补充”只用于放不适合单独加字段、但确实值得保留的信息，避免变成杂乱流水账。

## 使用方式

- 当当前互动对象明确时，直接读取对应人物文件全文。
- 人物文件用于给悠酱提供稳定的人物认知，不需要再从历史碎片中检索。
- 当前阶段先保证人物页可持续更新、可阅读、可手动修正。

## 第一阶段范围

- 只做 `people`，不单独设计 `event` 长期记忆。
- 先解决“记住人”和“按人读取”。
- 后续如果人物数量变多，再考虑补充更复杂的索引或检索能力。

## 技术实现

- 更新记忆时有一个更新 Agent，有一个审查 Agent，审查 Agent 会根据一定的记忆修改规则判断记忆更改是否合规。
