# 需求

## Action 运行记录与完成结算能力

### 背景

当前 world engine 更偏向“选择一个 Action 后立即执行副作用，再等待一段时间”。这种模式难以支撑做饭、种菜、钓鱼、打工等需要“开始行动、等待过程、完成结算”的能力。

例如“做饭”不应该只是一次即时状态变化，而应该包含：

- 开始做饭时选择食材，并记录角色正在做饭。
- 等待做饭耗时。
- 行动完成后，根据开始时选择的食材结算产出。
- 将产出的食物写入角色状态，并记录这次行动结果。

### 目标

为 world engine 增加 Action 的运行记录与完成结算能力，让 Action 可以拥有清晰的开始过程和完成结果。

核心目标：

- Action 开始时可以写入一条可追溯的运行记录。
- Redis 保存当前正在运行的 Action，用于恢复等待状态。
- MongoDB 保存本次 Action 从开始到完成的事实记录。
- Action 完成时可以基于开始时保存的上下文执行结算。
- 完成结算后产生 `completionEvent`，进入下一次 tick。

### 典型流程

以“做饭”为例：

```text
开始 Action
  -> 写入 runningAction：做饭
  -> 选择食材
  -> 在 MongoDB 写入一条 Action 运行记录，表示角色正在做饭
  -> 等待 duration
  -> Action 完成
  -> 执行完成结算
  -> 根据开始时选择的食材计算产出食物
  -> 更新角色背包或其他实时状态
  -> 可选择更新刚才写入 MongoDB 的 Action 运行记录
  -> 产生 completionEvent
  -> 进入下一次 tick
```

### 状态边界

- Redis 是角色实时状态的真相源，保存当前正在运行的 Action。
- MongoDB 保存 Action 运行过程的事实记录，用于追溯、记忆、日记和调试。
- 角色背包、体力、饱腹、心情、金币等实时状态仍以 Redis 为准。
- MongoDB 中的 Action 运行记录不作为角色实时状态的真相源。

### 验证场景

优先使用“做饭”验证该能力。

做饭需要覆盖的关键点：

- 开始时选择并保存食材。
- 完成时不能重新选择食材，而是读取开始时保存的食材。
- 完成结算时根据食材产出料理。
- 产出的料理进入背包。
- 运行记录能体现开始、完成和产出结果。

### 暂不处理范围

第一版暂不处理：

- 通用世界事件队列。
- 在完成结算流程中处理 Action 中断与取消，Action 中断由独立能力设计。
- 失败补偿和回滚。
- 复杂配方系统。
- 多角色并行行动。
- 场景状态系统。
- 大范围重构现有 Action。

## Action 中断能力

### 背景

部分运行中的 Action 需要允许被外部输入打断。典型场景是：悠酱半夜正在睡觉时，用户发送了强烈叫醒消息，希望她醒来聊天。

这个能力不应写成只服务睡觉的特殊逻辑，而应设计成通用的 Action 中断机制：运行中的 Action 可以声明自己是否允许被某类原因打断，以及被打断后如何结算。

第一版只实现“用户强烈消息打断睡觉”这一条规则。

### 目标

为 world engine 增加 Action 中断能力，让外部输入可以在规则允许时中断当前 `runningAction`。

核心目标：

- 支持对当前 `runningAction` 发起中断请求。
- 每个 Action 可以声明自己的中断策略。
- 中断请求需要包含来源和原因，如用户消息触发的叫醒请求。
- world engine 负责判断当前 Action 是否允许被该请求打断。
- 被打断后需要执行中断结算，更新实时状态并记录 Action 运行结果。
- message 模块只发起中断请求，不直接修改睡眠、体力、运行态等 world 状态。

### 典型流程

以“用户半夜叫醒睡觉中的悠酱”为例：

```text
用户发送消息
  -> message 模块读取当前 runningAction
  -> 当前 runningAction 是 Sleep
  -> 判断用户消息是否有强烈叫醒意图
  -> 如果没有强烈叫醒意图，继续按睡眠中的消息策略处理
  -> 如果有强烈叫醒意图，向 world engine 发起中断请求
  -> world engine 检查 Sleep 是否允许 user_wake_request 中断
  -> 执行 Sleep 的中断结算
  -> 更新 Action 运行记录为 interrupted
  -> 清理 Redis runningAction
  -> 更新角色被叫醒后的实时状态
  -> 返回“刚被叫醒”的聊天上下文
  -> message 模块继续生成聊天回复
```

### 通用设计

Action 中断请求需要至少表达：

- `source`：中断来源，如 `user_message`、`world_event`、`system`。
- `reason`：中断原因，如 `user_wake_request`。
- `message` 或上下文：触发中断的外部输入内容。
- `requestedAt`：中断请求发生时间。

Action 运行记录需要支持中断状态：

- `running`：正在运行。
- `completed`：正常完成。
- `interrupted`：运行中被打断。
- `failed`：运行失败。
- `cancelled`：被取消。

第一版可以只实际使用 `running`、`completed`、`interrupted`。

### 第一版范围

第一版只支持：

- 当前 `runningAction` 是 `Sleep`。
- 用户消息具有强烈叫醒意图。
- 中断原因是 `user_wake_request`。
- 中断来源是 `user_message`。
- 中断后不恢复原来的 Sleep。
- 中断后将“刚被叫醒”的上下文传给消息回复流程。

强烈叫醒意图可以先以规则判断为主，例如：

- 醒醒。
- 快醒醒。
- 起来。
- 别睡了。
- 有急事。
- 现在醒一下。
- 我需要你。

### 中断结算

Sleep 被用户叫醒后，不应等同于正常起床。

中断结算需要考虑：

- 根据已睡时长恢复部分体力。
- 饱腹可以轻微下降。
- 心情可能轻微下降。
- 清理当前 `runningAction`。
- 将 Action 运行记录标记为 `interrupted`。
- 记录中断原因和触发消息。
- 聊天上下文应明确“悠酱刚刚正在睡觉，被用户强烈叫醒”。

### 暂不处理范围

第一版暂不处理：

- 打断做饭、钓鱼、打工等其他 Action。
- 任意消息都能叫醒。
- 完全由 LLM 判断是否中断。
- 中断后恢复原 Action。
- 复杂中断优先级系统。
- 多个中断请求并发处理。

## 主动消息触发能力

### 背景

当前消息交互主要由用户发起，悠酱被动回复。为了让悠酱更像生活在世界中的角色，world engine 需要支持在合适的时机主动发送消息。

主动消息不是普通系统通知，而是悠酱基于自己的行动、状态、计划或生活事件，主动向用户分享、提醒或表达情绪。

### 目标

为 world engine 增加主动消息触发能力，让 Action 完成、状态异常、计划进展等事件在必要时可以触发一条发给用户的消息。

核心目标：

- Action 完成后可以选择性触发主动消息。
- 是否发送主动消息优先由规则判断，不默认每次 Action 完成都调用 LLM。
- 当规则判断需要发送时，再调用 LLM 生成自然的悠酱口吻消息。
- 主动消息需要记录，避免重复发送、频率过高或缺少追溯。
- 主动消息与 `completionEvent` 分离：`completionEvent` 服务下一次 tick，主动消息服务用户沟通。

### 典型流程

以“做饭完成”为例：

```text
Action 完成
  -> 执行完成结算
  -> 产出料理
  -> 产生 completionEvent，供下一次 tick 使用
  -> 规则判断这次结果是否值得主动告诉用户
  -> 如果不需要发送，流程结束
  -> 如果需要发送，构造主动消息上下文
  -> 调用 LLM 生成自然语言消息
  -> 通过 message 模块发送给用户
  -> 记录主动消息发送结果
```

### 触发边界

第一版可以优先支持 Action 完成后的主动消息。

适合触发主动消息的情况：

- 做饭完成并产出了料理。
- 钓鱼结束并获得了有趣或稀有的鱼获。
- 打工结束并结算工资。
- 睡前、醒来、完成计划等具有生活节奏意义的节点。
- 角色体力、饱腹或心情明显异常。
- 用户之前明确表达“记得告诉我”之类的期待。

通常不需要触发主动消息的情况：

- 普通移动完成。
- 发呆、短暂停留等低信息量 Action。
- 刚刚已经主动发送过消息。
- 当前处于安静时段。
- 当天主动消息次数已经过多。

### LLM 调用原则

- 是否发送主动消息，不完全交给 LLM 判断。
- 触发规则先判断是否值得发送。
- 只有确定需要发送时，才调用 LLM 生成具体文案。
- LLM 负责“怎么说”，规则负责“要不要说”。

### 状态边界

- world engine 负责判断主动消息触发时机，并提供消息上下文。
- message 模块负责生成或发送面向用户的消息。
- 主动消息发送记录需要可追溯，用于节流、去重和后续记忆整理。
- 主动消息不应直接修改角色实时状态，除非后续明确设计对应业务规则。

### 暂不处理范围

第一版暂不处理：

- 所有 Action 完成后都自动触发主动消息。
- 完全由 LLM 判断是否发送。
- 复杂的用户偏好系统。
- 多渠道主动消息策略。
- 基于大规模历史的个性化推送策略。

## 定时事项与未完成惩罚能力

### 背景

部分事情不是角色临时选择的普通 Action，而是工作日或特定时间必须面对的既定事项。典型场景是：工作日上课是悠酱的日常义务，如果她没有完成上课事项，应该承受一定惩罚。

上课 Action 负责表达“她怎么上课”，定时事项负责表达“她应不应该上课，以及没上课会怎样”。

### 目标

为 world engine 增加定时事项与未完成惩罚能力，让世界可以向角色提出固定时间段的生活要求，并在角色错过事项时进行规则结算。

核心目标：

- 支持定义工作日或特定时间段内应完成的事项。
- tick 决策时能感知当前正在进行或即将开始的定时事项。
- 定时事项可以声明完成条件。
- 当事项结束后仍未完成时，执行未完成惩罚。
- 未完成事项需要记录为可追溯事实，用于记忆、日记和后续决策。
- 惩罚由 world engine 规则结算，不交给 LLM 自行决定。

### 典型流程

以“工作日上午课”为例：

```text
工作日 09:00
  -> world engine 识别上午课事项开始
  -> tick 决策上下文包含：当前有上午课事项
  -> 悠酱可以选择去学校上课或继续上课
  -> 如果在 09:00-12:00 内执行过 Study_At_School，上午课事项完成
  -> 到 12:00 时检查事项完成状态
  -> 如果未完成，执行未完成惩罚
  -> 记录“上午课未完成”的事实
```

### 上课事项

第一版可以定义两个上课事项：

- 工作日上午课：工作日 `09:00-12:00`。
- 工作日下午课：工作日 `14:00-16:00`。

完成条件第一版保持简单：

- 在对应时间段内执行过 `Study_At_School`，即视为完成该课程事项。
- 整个时间段内都没有执行 `Study_At_School`，视为未完成。

后续可以再扩展迟到、早退、部分完成、请假、状态过差导致缺席等更细规则。

### 未完成惩罚

第一版惩罚应保持克制、明确、可追溯。

例如：

- 未完成上午课：心情下降，并记录“上午课没去上，有点不安”。
- 未完成下午课：心情下降，并记录“下午课错过了，感觉有点愧疚”。

惩罚需要通过明确规则结算，不应只依赖 LLM 生成叙事。

### 与 Action 的关系

- `Study_At_School` 仍然是普通 Action，负责执行上课行为本身。
- 定时事项负责判断上课是否是当前应该完成的事项。
- 定时事项负责在时间段结束后检查是否完成。
- Action 不应自己承担“工作日必须上课、没上课要惩罚”的全部规则。

### 暂不处理范围

第一版暂不处理：

- 复杂课程表。
- 迟到、早退、部分完成的细分评分。
- 请假系统。
- 连续缺席的累计惩罚。
- 用户指令导致缺席时的责任归因。
- 打工班次、吃饭时间、睡觉时间等其他定时事项。
