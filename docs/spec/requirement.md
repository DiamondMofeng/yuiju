## 移动生命周期能力

### 背景

当前移动类 Action 更像普通等待型 Action：开始执行时立即把角色位置更新到目的地，然后等待一段时间。这会导致角色在移动过程中看起来已经到达目的地，无法表达“正在从 A 前往 B”的状态。

移动能力不只是某几个 Action 的局部重构，而是 world engine 需要补充的一种生命周期能力。它需要表达出发地、目的地、移动中状态、到达结算，以及对状态查询和页面展示的影响。

### 目标

为 world engine 增加移动生命周期能力，让移动 Action 可以清晰表达“开始移动、移动中、到达完成”的过程。

核心目标：

- 移动开始时记录出发地和目的地。
- 移动过程中不提前把 `location` 更新为目的地。
- Redis 的 `runningAction` 保存移动中上下文，用于恢复等待状态和展示移动中状态。
- 移动完成时再更新角色 `location`。
- 移动完成时结算体力、饱腹等移动消耗。
- MongoDB behavior Episode 能体现移动从 `running` 到 `completed`。
- 状态查询和页面展示可以表达“正在从 A 前往 B”。

### 典型流程

以“从家去学校”为例：

```text
开始移动
  -> 读取当前位置作为 from
  -> 读取目标位置作为 to
  -> 写入 runningAction：移动中，from 家，to 学校
  -> 在 MongoDB 写入一条 running behavior Episode
  -> 等待移动耗时
  -> 移动完成
  -> 执行 completionEvent
  -> 更新 location 为学校
  -> 结算体力、饱腹等移动消耗
  -> 更新 MongoDB behavior Episode 为 completed
  -> 清理 Redis runningAction
  -> 产生 eventDescription
  -> 进入下一次 tick
```

### 状态边界

- Redis 是角色实时状态的真相源，移动中状态由 `runningAction` 表达。
- `location` 只表示角色当前已经处于的位置，不表示移动目的地。
- 移动完成前，不应把 `location` 提前更新为目的地。
- MongoDB behavior Episode 用于记录移动事实，不作为实时位置真相源。
- 页面或状态查询需要展示移动中状态时，应从 `location` 和 `runningAction` 派生展示，不新增第二份位置真相源。

### 第一版范围

第一版只处理现有移动类 Action：

- 移动开始时记录 `from` 和 `to`。
- 移动中保留原 `location`。
- 移动完成时更新 `location`。
- 移动完成时结算移动消耗。
- 移动完成后返回可作为下一次 tick 上下文的 `eventDescription`。

### 暂不处理范围

第一版暂不处理：

- 移动中断、取消或改道。
- 移动中随机事件。
- 多段路线和路线选择。
- 交通方式系统。
- 多角色并行移动。
- 场景状态系统。

## Action 中断能力

### 背景

部分运行中的 Action 需要允许被外部输入打断。典型场景是：悠酱半夜正在睡觉时，用户发送了强烈叫醒消息，希望她醒来聊天。

这个能力不应写成只服务睡觉的特殊逻辑，而应设计成通用的 Action 中断机制：运行中的 Action 可以声明自己是否允许被某类原因打断，以及被打断后如何结算。

第一版只实现“用户强烈消息打断睡觉”这一条规则。

### 目标

为 world engine 增加 Action 中断能力，让外部输入可以在规则允许时中断当前 `runningAction`。

核心目标：

- 支持对当前 `runningAction` 发起中断请求。
- 每个 Action 可以声明自己的中断策略。
- 中断请求需要包含来源和原因，如用户消息触发的叫醒请求。
- world engine 负责判断当前 Action 是否允许被该请求打断。
- 被打断后需要执行中断结算，更新实时状态并记录 Action 运行结果。
- message 模块只发起中断请求，不直接修改睡眠、体力、运行态等 world 状态。

### 典型流程

以“用户半夜叫醒睡觉中的悠酱”为例：

```text
用户发送消息
  -> message 模块读取当前 runningAction
  -> 当前 runningAction 是 Sleep
  -> 判断用户消息是否有强烈叫醒意图
  -> 如果没有强烈叫醒意图，继续按睡眠中的消息策略处理
  -> 如果有强烈叫醒意图，向 world engine 发起中断请求
  -> world engine 检查 Sleep 是否允许 user_wake_request 中断
  -> 执行 Sleep 的中断结算
  -> 更新 Action 运行记录为 interrupted
  -> 清理 Redis runningAction
  -> 更新角色被叫醒后的实时状态
  -> 返回“刚被叫醒”的聊天上下文
  -> message 模块继续生成聊天回复
```

### 通用设计

Action 中断请求需要至少表达：

- `source`：中断来源，如 `user_message`、`world_event`、`system`。
- `reason`：中断原因，如 `user_wake_request`。
- `message` 或上下文：触发中断的外部输入内容。
- `requestedAt`：中断请求发生时间。

Action 运行记录需要支持中断状态：

- `running`：正在运行。
- `completed`：正常完成。
- `interrupted`：运行中被打断。
- `failed`：运行失败。
- `cancelled`：被取消。

第一版可以只实际使用 `running`、`completed`、`interrupted`。

### 第一版范围

第一版只支持：

- 当前 `runningAction` 是 `Sleep`。
- 用户消息具有强烈叫醒意图。
- 中断原因是 `user_wake_request`。
- 中断来源是 `user_message`。
- 中断后不恢复原来的 Sleep。
- 中断后将“刚被叫醒”的上下文传给消息回复流程。

强烈叫醒意图可以先以规则判断为主，例如：

- 醒醒。
- 快醒醒。
- 起来。
- 别睡了。
- 有急事。
- 现在醒一下。
- 我需要你。

### 中断结算

Sleep 被用户叫醒后，不应等同于正常起床。

中断结算需要考虑：

- 根据已睡时长恢复部分体力。
- 饱腹可以轻微下降。
- 心情可能轻微下降。
- 清理当前 `runningAction`。
- 将 Action 运行记录标记为 `interrupted`。
- 记录中断原因和触发消息。
- 聊天上下文应明确“悠酱刚刚正在睡觉，被用户强烈叫醒”。

### 暂不处理范围

第一版暂不处理：

- 打断做饭、钓鱼、打工等其他 Action。
- 任意消息都能叫醒。
- 完全由 LLM 判断是否中断。
- 中断后恢复原 Action。
- 复杂中断优先级系统。
- 多个中断请求并发处理。

## 定时事项与未完成惩罚能力

### 背景

部分事情不是角色临时选择的普通 Action，而是工作日或特定时间必须面对的既定事项。典型场景是：工作日上课是悠酱的日常义务，如果她没有完成上课事项，应该承受一定惩罚。

上课 Action 负责表达“她怎么上课”，定时事项负责表达“她应不应该上课，以及没上课会怎样”。

### 目标

为 world engine 增加定时事项与未完成惩罚能力，让世界可以向角色提出固定时间段的生活要求，并在角色错过事项时进行规则结算。

核心目标：

- 支持定义工作日或特定时间段内应完成的事项。
- tick 决策时能感知当前正在进行或即将开始的定时事项。
- 定时事项可以声明完成条件。
- 当事项结束后仍未完成时，执行未完成惩罚。
- 未完成事项需要记录为可追溯事实，用于记忆、日记和后续决策。
- 惩罚由 world engine 规则结算，不交给 LLM 自行决定。

### 典型流程

以“工作日上午课”为例：

```text
工作日 09:00
  -> world engine 识别上午课事项开始
  -> tick 决策上下文包含：当前有上午课事项
  -> 悠酱可以选择去学校上课或继续上课
  -> 如果在 09:00-12:00 内执行过 Study_At_School，上午课事项完成
  -> 到 12:00 时检查事项完成状态
  -> 如果未完成，执行未完成惩罚
  -> 记录“上午课未完成”的事实
```

### 上课事项

第一版可以定义两个上课事项：

- 工作日上午课：工作日 `09:00-12:00`。
- 工作日下午课：工作日 `14:00-16:00`。

完成条件第一版保持简单：

- 在对应时间段内执行过 `Study_At_School`，即视为完成该课程事项。
- 整个时间段内都没有执行 `Study_At_School`，视为未完成。

后续可以再扩展迟到、早退、部分完成、请假、状态过差导致缺席等更细规则。

### 未完成惩罚

第一版惩罚应保持克制、明确、可追溯。

例如：

- 未完成上午课：心情下降，并记录“上午课没去上，有点不安”。
- 未完成下午课：心情下降，并记录“下午课错过了，感觉有点愧疚”。

惩罚需要通过明确规则结算，不应只依赖 LLM 生成叙事。

### 与 Action 的关系

- `Study_At_School` 仍然是普通 Action，负责执行上课行为本身。
- 定时事项负责判断上课是否是当前应该完成的事项。
- 定时事项负责在时间段结束后检查是否完成。
- Action 不应自己承担“工作日必须上课、没上课要惩罚”的全部规则。

### 暂不处理范围

第一版暂不处理：

- 复杂课程表。
- 迟到、早退、部分完成的细分评分。
- 请假系统。
- 连续缺席的累计惩罚。
- 用户指令导致缺席时的责任归因。
- 打工班次、吃饭时间、睡觉时间等其他定时事项。
