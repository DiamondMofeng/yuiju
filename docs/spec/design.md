# 技术方案

## 目标

长期记忆第一阶段只聚焦 `people`，不做事件检索型知识库，也不做向量检索。

目标是：

- 每个朋友对应一个独立的人物记忆文件
- 当前互动对象明确时，直接读取这个人的完整记忆文件
- 记忆文件可以被悠酱持续更新
- 更新过程有审查，避免把猜测写成事实，或把不稳定信息写进长期记忆

## 存储形式

人物长期记忆使用 Markdown 文件存储。

目录结构：

```text
memory/
  people/
    123456789.md
    987654321.md
```

每个人物对应一个独立文件，文件名使用稳定的人物标识。

第一版中，`personId` 直接使用 QQ 号：

- 文件名使用 QQ 号，如 `123456789.md`
- function tool 读取人物记忆时，也使用 QQ 号作为唯一入参
- 人物显示名、称呼、群名片变化时，不影响文件路径和人物标识

## 人物记忆文件格式

第一版人物记忆文件采用固定标题的 Markdown 结构，不做复杂模块划分。

文件结构固定如下：

```md
---
last_updated_at: 2026-04-18T21:30:00+08:00
---

# 小林

## 称呼
小林 / 小林同学

## 喜好
安静聊天，甜食，放学后慢慢散步

## 雷区
被催得很急，在很多人面前起哄她

## 最近在忙什么
最近在准备考试，晚上也会补作业，状态有点累。

## 悠酱对她的态度
觉得她是个温柔但不太会主动表达的人，愿意慢慢靠近她，也会下意识照顾她的情绪。

## 最近一次值得记住的互动
昨天群里聊到考试压力时，她少见地主动说自己最近有点撑不住，悠酱安慰了她几句，她后来回了一个“谢谢”。

## 其他补充
主要在群聊里互动，偶尔也会私聊。
```

约束：

- front matter 第一版只包含一个字段：`last_updated_at`
- 一级标题 `#` 表示人物显示名。
- 二级标题固定为以下 7 个，不能增删、不能改名、不能调整顺序：
  - `称呼`
  - `喜好`
  - `雷区`
  - `最近在忙什么`
  - `悠酱对她的态度`
  - `最近一次值得记住的互动`
  - `其他补充`
- 每个标题下只放纯文本内容，不嵌套列表、表格或更多子标题。
- 如果某个字段暂时没有内容，也保留标题，正文写 `（暂无）`。
- `last_updated_at` 只表示“人物记忆文件最后一次成功写回的时间”，不表示最后一次互动时间。

## 读取方案

第一版读取逻辑保持简单，不做搜索和检索，只按人直接读取。

读取方式：

- 上层在互动场景中明确当前对象是谁。
- 根据 `personId` 定位到对应的 Markdown 文件。
- 直接读取整份文件全文，作为人物长期记忆输入。

第一版不要求把 Markdown 解析成额外的数据结构，模块内部可直接以全文字符串为主。

如果文件不存在，则视为该人物尚未建立长期记忆。

其中：

- `personId` 直接使用 QQ 号
- 人物记忆模块内部不负责把自然语言称呼解析成 `personId`
- 如果调用方当前只知道名字或称呼，应先通过人物目录能力确认目标人物
- 如果无法确认明确的 `personId`，则人物记忆模块不读取、不创建、也不更新人物记忆文件

## 新建人物记忆文件

人物记忆文件不是一有互动就立即创建，而是要达到一定互动门槛后才新建。

创建条件：

- 当前互动对象身份明确，能够拿到稳定的 `personId`
- 这次互动已经形成了足够的人物认知价值，值得建立长期人物记忆

不创建的情况：

- 只是群聊里路过的一句发言
- 没有形成实际互动
- 没有可稳定沉淀的人物认知
- 当前对象身份不明确，无法确认 `personId`

第一版建议采用保守策略：只有当更新 Agent 判断这次互动已经足以支撑建立人物页时，才创建新文件。

## 更新触发

第一版更新模块不负责切分消息窗口，也不负责判断原始消息流应该如何归档。

更新模块只接收：

- 旧的人物记忆 Markdown
- 一段已经整理好的互动材料文本

推荐的触发方式是：

- 上层在一次人物互动结束后，整理出一段互动材料
- 调用人物记忆更新模块
- 模块内部判断这次是否值得更新长期记忆

不建议按单条消息直接更新人物记忆文件。

## 更新时机

人物记忆文件的更新时机分为“候选时机”和“真正更新时机”两层。

### 候选时机

候选时机指“可以考虑触发一次人物记忆更新判断”的时刻。

第一版推荐以“一段针对某个人的互动结束”作为候选时机。

这里的“一段互动”不要求由记忆模块自己识别。更推荐由上层整理好一段互动材料后，再调用记忆模块。

候选时机的典型形式：

- 一次和某个人相关的私聊互动结束后
- 一次群聊中围绕某个人形成的互动片段结束后
- 上层完成一次聊天窗口归档后

如果是“悠酱和群里多人聊天”，则允许同一段群聊材料拆成多份人物更新候选。

约束：

- 每一份人物更新候选都必须绑定一个明确的 `personId`
- 一段群聊材料可以产出 0 到 N 份人物更新候选
- 如果没有围绕某个明确人物形成稳定认知，则不产出对应候选

第一版不建议以下时机直接触发人物记忆更新：

- 每收到一条消息就更新
- 固定按时间批量重写所有人物记忆
- 读取人物记忆时顺手更新

### 真正更新时机

候选时机到来后，不代表一定会更新文件。

真正更新的条件是：

- 更新 Agent 判断这段互动带来了值得写入长期记忆的新信息
- 审查 Agent 审查通过

如果当前互动对象身份不明确，无法确认 `personId`，则不进入真正更新，也不新建人物记忆文件。

也就是说，推荐流程是：

1. 一段人物互动结束
2. 上层整理互动材料
3. 触发一次人物记忆更新判断
4. 只有判断为“值得更新”时，才进入提案、审查和写回流程

这意味着第一版的人物记忆更新策略不是“互动一结束就改文件”，而是“互动一结束就判断，只有必要时才改文件”。

### 这样设计的原因

不按单条消息更新，而是按一段互动结束后再判断，主要是为了：

- 避免信息过碎，导致长期记忆抖动
- 避免把一句随口的话误写成稳定认知
- 降低更新和审查频率
- 让 `最近在忙什么`、`悠酱对她的态度`、`最近一次值得记住的互动` 这类字段基于完整上下文形成阶段性结论

### 通常值得更新的情况

以下情况通常值得进入人物记忆更新判断，并有较大概率真正更新：

- 对方明确透露了近期近况
- 对方明确表达了稳定喜好或明显反感
- 这次互动让悠酱对这个人的态度形成了新的判断
- 出现了一次明显值得记住的互动
- 旧记忆中相关内容已经过期或明显不准确

### 通常应跳过的情况

以下情况通常应直接跳过，不进入真正更新：

- 只是普通寒暄
- 没有关于这个人的新信息
- 只是重复已有认知
- 只是群里顺手接一句，没有形成真正互动
- 信息模糊，证据不足，无法支撑长期记忆修改

## 更新流程

人物记忆更新采用“更新 Agent 提案 + 审查 Agent 审核 + 更新 Agent 执行”的流程。

完整流程如下：

1. 读取旧的人物记忆 Markdown。
2. 准备本次互动材料文本。
3. 更新 Agent 先给出“修改提案”，说明：
   - 要修改哪些标题
   - 每个标题准备改成什么内容
   - 为什么这样修改
4. 审查 Agent 审查这份修改提案是否合规。
5. 如果审查通过：
   - 更新 Agent 严格按照已通过的提案修改 Markdown
   - 输出更新后的完整 Markdown
   - 流程结束
6. 如果审查驳回：
   - 审查 Agent 给出驳回原因和修改意见
   - 更新 Agent 根据反馈调整提案
   - 再次提交审查
7. 如果超过最大迭代次数仍未通过，则放弃本次更新，保留旧文件不变。

约束：

- 审查通过前，更新 Agent 不直接修改人物记忆文件。
- 审查通过后，更新 Agent 只能执行已通过的提案，不能顺手修改其他内容。
- 提案中没有涉及的标题区块必须原样保留。
- 最大迭代次数建议为 2 到 3 轮，避免更新和审查来回拉扯。

## 更新 Agent 职责

更新 Agent 负责：

- 读取旧的人物记忆 Markdown
- 理解本次互动材料
- 判断这次是否值得更新人物长期记忆
- 产出文本形式的修改提案
- 在审查通过后，根据提案生成新的完整 Markdown
- 在审查驳回后，根据反馈调整提案

更新 Agent 的提案应包含三部分信息：

- 要修改的标题
- 标题修改后的完整新内容
- 修改原因

这里的“修改后内容”必须是对应标题下的完整正文，而不是“增加一句”或“删除一句”的局部指令。

## 审查 Agent 职责

审查 Agent 负责审查“修改提案”，不直接修改人物记忆文件。

审查重点：

- 修改是否有明确依据，且依据来自本次互动材料
- 是否把猜测、印象或一次性信息写成了长期事实
- 是否错误修改了高门槛字段，如 `称呼`、`喜好`、`雷区`
- 是否对 `悠酱对她的态度` 做了过大的跳跃式修改
- 是否破坏了人物记忆文件的固定标题结构

审查结果只有两种：

- 通过
- 驳回，并给出具体问题与修改建议

## 字段更新规则

不同标题的更新门槛不同：

### 称呼

- 只有在互动中明确出现更稳定的新称呼时才更新。
- 不因为一次临时叫法就修改。

### 喜好

- 需要明确表达，或在多次互动中重复出现。
- 一次随口提到通常不足以写入长期记忆。

### 雷区

- 更新门槛最高。
- 需要明确的不喜欢、反感、回避或稳定负面反馈，才允许写入。

### 最近在忙什么

- 可以覆盖更新。
- 只保留当前阶段仍有效的近况。
- 过期内容应直接替换。

### 悠酱对她的态度

- 可以更新，但应保守，只允许小步调整。
- 不应因为一次普通互动就发生明显跳变。

### 最近一次值得记住的互动

- 只保留一条最新且确实值得记住的互动。
- 不追加为历史列表。

### 其他补充

- 只用于承接确实值得保留、但不适合放入其他标题的信息。
- 不能演变成杂项流水账。

## 不更新的场景

以下情况应优先跳过，不更新人物记忆文件：

- 只是普通寒暄，没有新信息
- 只是重复已有信息，没有形成新的稳定认知
- 无法确定互动对象是否就是当前人物
- 证据不足，不足以支持修改长期记忆
- 这次互动不构成“最近一次值得记住的互动”

## 模块边界

这一模块是相对独立的人物长期记忆模块。

它负责：

- 按人读取人物 Markdown
- 基于互动材料驱动双 Agent 更新流程
- 在审查通过后写回文件

它不负责：

- 原始消息采集
- 会话窗口切分
- 向量检索或全文搜索
- 旧记忆系统兼容

## Function Tool 设计

人物长期记忆对 LLM 暴露两个独立的 function tool：

- `listPersonMemoriesTool`
- `getPersonMemoryTool`

这两个 tool 与现有 `memorySearchTool` 分开设计，不放进同一个 tool 中。

原因：

- `memorySearchTool` 的职责是搜索 episode / diary / fact，心智模型是“查询结果列表”
- 人物记忆的职责是按人直接读取整份人物页，心智模型是“按主键读取完整文档”
- 如果把人物记忆读取能力混入 `memorySearchTool`，会让“搜索”和“按人读取”两种完全不同的能力混在一起，增加模型使用歧义

### listPersonMemoriesTool

职责：

- 列出当前已经存在人物记忆文件的人物目录

适用场景：

- 模型不知道当前有哪些人物记忆
- 当前对象不够明确，需要先确认候选人物
- 需要先查看人物目录，再决定读取谁的完整记忆

第一版建议无参调用。

返回内容建议包含最小必要信息：

- `personId`
- `displayName`
- `appellation`

其中：

- `personId` 使用 QQ 号
- `displayName` 用于展示人物名称
- `appellation` 用于辅助辨识人物当前称呼

`listPersonMemoriesTool` 只承担“列目录”的职责，不做语义搜索、排序召回或全文读取。

### getPersonMemoryTool

职责：

- 按 `personId` 读取对应人物记忆全文

入参：

- `personId`

其中：

- `personId` 直接使用 QQ 号
- 第一版只接受稳定 `personId`，不接受自然语言名字或称呼

返回内容：

- `personId`
- `displayName`
- `memoryMarkdown`

其中：

- `memoryMarkdown` 为该人物完整的 Markdown 记忆文件全文

`getPersonMemoryTool` 不负责：

- 模糊匹配人物
- 根据称呼猜测人物
- 语义搜索人物记忆
- 更新人物记忆

### 推荐调用方式

- 当当前对象明确，且已经知道 `personId` 时，直接调用 `getPersonMemoryTool`
- 当当前对象不够明确，或者只知道名字 / 称呼时，先调用 `listPersonMemoriesTool`
- 这两个 tool 都不应被当作“语义搜索工具”使用
